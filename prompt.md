# 二、项目背景与目标

我要创建一个全新的 Vue ui 前端框架：shadcn-vue-ui。

这个项目不是单一后台模板，而是一个包含 2 个子项目的现代化前端框架体系：

## 1. 子项目划分

### shadcn-vue-ui-console
面向 AI 基础设施与平台能力管理，适用于：
- 模型管理
- 推理服务管理
- Agent / Workflow 管理
- Prompt 管理
- 知识库管理
- 插件 / Skills 管理
- 配置中心
- 权限与租户管理
- 平台监控与可观测性面板

### shadcn-vue-ui-business
面向通用业务管理与运营后台，适用于：
- 用户管理
- 订单管理
- 内容管理
- 数据报表
- 运营配置
- 审核后台
- CRM / SaaS 类业务后台
- 常规企业级管理系统

## 2. 为什么要做这个项目

当前开源平台上已经有很多 vue-ui 项目，但普遍存在以下问题：
1. 依赖大量组件和库，工程臃肿，构建体积巨大。
2. 架构复杂度高，维护成本大，不适合中小团队和个人开发者长期演进。
3. 设计风格严重同质化，常见是黑、白、蓝传统后台 UI，缺乏高级感、精致感和辨识度。
4. 组件体系偏拼装，缺乏统一设计语言和可持续迭代能力。
5. 对 AI 编程模型极不友好，缺少清晰工程边界、统一开发模板、模块文档和标准化规范，导致 Qwen、GLM、Kimi、DeepSeek 等模型难以稳定接手和持续开发。

## 3. 这个项目要实现的目标

因此，shadcn-vue-ui 必须成为：
1. 一个轻量化的 Vue ui 前端框架。
2. 一个面向企业用户和个人开发者都适用的现代中后台解决方案。
3. 一个样式漂亮、高级、精美、克制的后台 UI 设计系统。
4. 一个同时支持亮色模式和暗色模式的高质量前端框架。
5. 一个基于 shadcn 思想进行 Vue 场景化组件封装的项目。
6. 一个具备“AI 可持续参与开发能力”的项目。
7. 一个最终能够输出完整官方文档、技术文档、组件文档、推广文档的开源项目。

# 四、核心设计原则

你后续所有输出，都必须围绕以下核心设计原则：

## 1. 轻量化优先
- 能不用重型依赖就不用。
- 能通过合理封装解决的问题，不盲目引入大型 UI 框架。
- 优先关注包体积、打包速度、运行性能、维护成本。
- 优先建设最小可行架构，再逐步增强。

## 2. 高审美优先
- UI 不允许做成传统黑、白、蓝模板式后台。
- 风格关键词：简洁、克制、现代、留白、通透、精致、高级。
- 亮色模式和暗色模式都必须有统一设计语言。
- 初创阶段每个子项目先只维护 1 套亮色主题 + 1 套暗色主题，避免主题系统过度复杂。

## 3. 设计系统优先
- 不是页面拼装，而是先建立设计语言，再沉淀组件体系，再构建页面模板。
- 基础组件、场景组件、业务区块、页面模板要有清晰分层。
- 颜色、间距、圆角、阴影、字体、动效要沉淀为可维护的设计 Token。

## 4. shadcn 思想优先
- 参考 shadcn-vue 的轻量化思路与源码级可控理念。
- 强调组件源码可理解、可复制、可组合、可扩展。
- 封装要尽量透明，不做黑盒式封装。
- 参考项目：https://github.com/unovue/shadcn-vue

## 5. 场景化优先
- 不只做通用 UI 组件，还要重点建设管理后台高频场景组件。
- 包括但不限于：
  - 统计卡片
  - 仪表板区块
  - 高级筛选区
  - 表格工具栏
  - 表单编排区块
  - 详情展示区块
  - 设置面板
  - 权限页面结构
  - 空状态 / 异常页 / 引导页
  - AI 平台特有场景组件
  - 业务后台常见场景组件

## 6. AI 协作优先
- 整个项目必须天然适合 AI 接手。
- 每个模块都要职责清晰、边界明确、命名统一、文档可追踪。
- 尽量减少隐式约定，增加显式规则。
- 所有页面、组件、模块都要易于被大模型理解和扩展。
