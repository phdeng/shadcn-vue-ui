# 决策架构工作流 - 技术参考

**模块：** BMM (BMAD 方法模块)
**类型：** 解决方案工作流

---

## 概述

决策架构工作流是对 BMAD 方法中如何进行架构决策的全新重新构想。该工作流不采用模板驱动的文档，而是促进智能对话以产生 **以决策为中心的架构文档**，优化实施期间防止 AI Agent 冲突。

---

## 核心哲学

**问题：** 当多个 AI Agent 实施系统的不同部分时，它们会做出冲突的技术决策，导致不兼容的实施。

**解决方案：** 一个"一致性合同"，提前记录所有关键的技术决策，确保每个 Agent 遵循相同的模式并使用相同的技术。

---

## 主要特性

### 1. 启动模板智能 ⭐ 新增

- 发现相关启动模板 (create-next-app, create-t3-app 等)
- 选择模板时考虑 UX 需求 (动画、可访问性等)
- 搜索当前 CLI 选项和默认值
- 记录启动模板作出的决策
- 围绕启动基础进行剩余的架构决策
- 首个实施故事变为"使用启动命令初始化"

### 2. 自适应引导

- 根据用户技能水平调整对话风格 (初学者/中级/专家)
- 专家获得快速、技术性的讨论
- 初学者获得教育和复杂性的保护
- 每个人都产生相同高质量的输出

### 3. 动态版本验证

- 绝不信任硬编码版本号
- 使用 WebSearch 查找当前稳定版本
- 在对话期间验证版本
- 仅记录已验证的当前版本

### 4. 智能发现

- 无刚性项目类型模板
- 分析 PRD 以确定哪些决策对本项目重要
- 使用决策和模式的知识库
- 无限扩展至项目类型

### 5. 协作决策制定

- 促进每个关键决策的讨论
- 提供带有权衡的选项
- 集成高级引导以获得创新方法
- 确保决策的一致性和兼容性

### 6. 一致输出

- 对话期间的结构化决策收集
- 基于收集的决策进行严格文档生成
- 根据硬性要求验证
- 为 AI Agent 消费优化

---

## 工作流结构

```
步骤 0：验证工作流并提取项目配置
步骤 0.5：验证工作流排序
步骤 1：加载 PRD (含 FRs/NFRs) 并理解项目上下文
步骤 2：发现和评估启动模板 ⭐ 新增
步骤 3：调整引导风格并识别剩余决策
步骤 4：促进协作决策制定 (含版本验证)
步骤 5：处理横切关注点
步骤 6：定义项目结构和边界
步骤 7：设计新颖架构模式 (如需要) ⭐ 新增
步骤 8：定义实施模式以防止 Agent 冲突
步骤 9：验证架构一致性
步骤 10：生成决策架构文档 (含初始化命令)
步骤 11：验证文档完整性
步骤 12：最终审查并更新工作流状态
```

---

## 此工作流中的文件

- **workflow.yaml** - 配置和元数据
- **instructions.md** - 自适应引导流程
- **decision-catalog.yaml** - 所有架构决策的知识库
- **architecture-patterns.yaml** - 从需求中识别的通用模式
- **pattern-categories.csv** - 教授 LLM 需要定义什么的模式原则
- **checklist.md** - 输出文档的验证要求
- **architecture-template.md** - 最终文档的严格格式

## 与旧版 architecture 的不同之处

| 方面                 | 旧工作流                                     | 新工作流                                        |
| -------------------- | -------------------------------------------- | ----------------------------------------------- |
| **方法**             | 模板驱动                                     | 对话驱动                                        |
| **项目类型**         | 11 种刚性类型，含 22+ 文件                   | 智能发现的无限灵活性                            |
| **用户交互**         | 输出章节含"继续？"                           | 协作决策引导                                    |
| **技能适配**         | 一刀切方案                                   | 适配至初学者/中级/专家                          |
| **决策制定**         | 过程后期 (步骤 5)                            | 前置并集中关注                                  |
| **输出**             | 多个文档包括虚假技术规范                     | 单一的以决策为中心的架构                        |
| **时间**             | 混乱且缓慢                                   | 30-90 分钟，取决于技能水平                      |
| **引导**             | 从未使用                                     | 在决策点集成                                    |

## 期望输入

- **PRD** (产品需求文档)，包括：
  - 功能需求
  - 非功能需求
  - 性能和合规需求

- **UX 规范** (可选但有价值)，包括：
  - 界面设计和交互模式
  - 可访问性需求 (WCAG 等级)
  - 动画和转换需求
  - 平台特定 UI 需求
  - 交互的性能期望

---

## 输出文档

单个 `architecture.md` 文件包含：

- 执行摘要 (2-3 句话)
- 项目初始化命令 (如使用启动模板)
- 带已验证版本和史诗映射的决策摘要表
- 完整的项目结构
- 集成规范
- AI Agent 的一致性规则

## 新型模式设计工作原理

步骤 7 处理需要被创造的独特或复杂模式：

### 1. 检测

工作流分析 PRD 中没有标准解决方案的概念：

- 新颖的交互模式 (例如，当 Tinder 不存在时的"滑动匹配")
- 复杂的多史诗工作流 (例如，"病毒式邀请系统")
- 独特的数据关系 (例如，Facebook 之前的"社交图")
- 新范式 (例如，Snapchat 之前的"临时消息")

### 2. 设计协作

工作流不只是选择技术，而是帮助设计解决方案：

- 确定要解决的核心问题
- 与用户探索不同的方法
- 记录组件如何交互
- 为复杂流程创建序列图
- 使用引导发现创新解决方案

### 3. 文档

新型模式成为架构的一部分，包括：

- 模式名称和用途
- 组件交互
- 数据流图
- 受影响的史诗/故事
- Agent 实施指导

### 4. 示例

```
PRD："用户可以创建重叠成员资格的朋友'圈子'"
↓
工作流检测：这是一个新颖的社交结构模式
↓
与用户设计：圈子成员模型、权限级联、UI 模式
↓
文档："圈子模式"含组件设计和数据流
↓
所有 Agent 都了解如何一致地实施圈子相关功能
```

## 实施模式工作原理

步骤 8 通过定义一致性模式防止 Agent 冲突：

### 1. 核心原则

> "任何多 Agent 可能以不同方式做出相同决策的情况，都是需要捕捉的模式"

LLM 询问："Agent 遇到需要猜测什么情况？"

### 2. 模式类别 (原则，非规定)

- **命名**：事物如何命名 (API、数据库字段、文件)
- **结构**：事物如何组织 (文件夹、模块、层)
- **格式**：数据如何格式化 (JSON 结构、响应)
- **通信**：组件如何通信 (事件、消息、协议)
- **生命周期**：状态如何变化 (工作流、转换)
- **位置**：事物去往何处 (URL、路径、存储)
- **一致性**：横切关注点 (日期、错误、日志)

### 3. LLM 智能

- 使用原则识别超出 7 个类别的模式
- 确定所选技术的特定模式重要性
- 仅询问可能引起冲突的模式
- 跳过技术选择确定的明显模式

### 4. 示例

```
选择的技术：REST API + PostgreSQL + React
↓
LLM 识别需求：
- REST：URL 结构、响应格式、状态码
- PostgreSQL：表命名、列命名、外键模式
- React：组件结构、状态管理、测试位置
↓
与用户进行每个方面的引导
↓
作为实施模式记录在架构中
```

## 启动模板工作原理

当工作流检测到有启动模板的项目类型时：

1. **发现**：根据 PRD 搜索相关启动模板
2. **调查**：查找当前 CLI 选项和默认值
3. **展示**：向用户展示启动模板提供的内容
4. **集成**：将启动模板决策记录为"由启动模板提供"
5. **继续**：仅询问启动模板未做出的决策
6. **文档**：在架构中包含确切的初始化命令

### 示例流程

```
PRD 说明："带认证的 Next.js Web 应用"
↓
工作流找到：create-next-app 和 create-t3-app
↓
用户选择：create-t3-app (包含认证设置)
↓
启动模板提供：Next.js、TypeScript、tRPC、Prisma、NextAuth、Tailwind
↓
工作流仅询问：数据库选择、部署目标、附加服务
↓
首个故事变为："npx create t3-app@latest my-app --trpc --nextauth --prisma"
```

## 用法

```bash
# 在您的 BMAD 启用项目中
workflow architecture
```

AI Agent 将：

1. 加载您的 PRD (含 FRs/NFRs)
2. 识别所需的关键决策
3. 促进每个决策的讨论
4. 生成全面的架构文档
5. 验证完整性

---

## 设计原则

1. **引导优于规定** - 引导用户做出好决策，而不是施加模板
2. **智能优于模板** - 使用 AI 理解而不是刚性结构
3. **决策优于细节** - 专注于防止 Agent 冲突，而不是实施细节
4. **适配优于统一** - 在确保质量输出的同时满足用户需求
5. **协作优于输出** - 对话与文档同等重要

---

## 对于开发者

此工作流假设：

- 单个开发者 + AI Agent (而非团队)
- 速度重要 (分钟级决策，而非天级)
- AI Agent 需要明确约束防止冲突
- 架构文档用于 Agent，而非人类

---

## 从 architecture 迁移

使用旧版 `architecture` 工作流的项目应：

1. 完成任何进行中的架构工作
2. 对新项目使用 `architecture`
3. 旧工作流仍可用但已弃用
