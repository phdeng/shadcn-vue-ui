# 路线修正 - 冲刺变更管理说明

## 工作流概述

路线修正工作流帮助处理冲刺期间出现的变更请求和问题，包括：
- 范围变更请求
- 需求澄清
- 技术决策调整
- 性能问题或缺陷

此工作流确保所有变更都经过评估、批准并进行沟通，以保持冲刺的完整性。

## 工作流步骤（6 步）

### 步骤 1：识别和分类变更

**操作**:
1. 识别导致变更请求的原因
2. 将其分类为以下之一：
   - 范围蠕变（添加功能）
   - 需求变更（改变现有需求）
   - 技术决策变更（架构或设计调整）
   - 实现问题（发现的技术障碍）
   - 性能问题（性能缺陷或改进机会）
   - 其他

**输出**:
- 变更分类和描述
- 变更来源（谁提出）
- 变更优先级初步评估

### 步骤 2：分析影响

**操作**:
1. 识别受影响的故事和史诗
2. 识别依赖关系（哪些其他故事会受到影响）
3. 评估工作量增加
4. 识别相关的技术或架构风险

**输出**:
- 受影响的项目清单
- 工作量影响评估
- 风险评估
- 可能的替代方案

### 步骤 3：制定决策

**决策选项**:

1. **接受变更**
   - 调整计划和冲刺目标
   - 可能推迟其他故事或调整优先级

2. **拒绝变更**
   - 记录理由
   - 建议替代方案或未来时间表

3. **推迟变更**
   - 将其添加到下一个冲刺的积压
   - 记录理由

4. **分解变更**
   - 将变更分成较小的增量部分
   - 在后续冲刺中逐步实现

**输出**:
- 最终决策
- 决策理由
- 批准人

### 步骤 4：更新计划

**操作**:
1. 更新受影响的故事
2. 更新冲刺优先级和目标
3. 更新冲刺状态文件
4. 如需要，重新估算故事

**输出**:
- 更新的冲刺计划
- 修改的故事描述
- 更新的冲刺状态文件

### 步骤 5：沟通和风险管理

**操作**:
1. 与所有利益相关者沟通变更
2. 解释决策和影响
3. 识别并记录相关风险
4. 制定风险缓解计划

**输出**:
- 沟通记录
- 风险评估和缓解计划
- 团队确认

### 步骤 6：跟踪和评估

**操作**:
1. 监控变更的实施
2. 跟踪实际影响与估计
3. 记录结果和教训
4. 更新历史记录用于未来改进

**输出**:
- 变更跟踪记录
- 最终结果评估
- 经验教训
- 对未来流程的改进建议

## 关键原则

1. **迅速决策** - 快速评估和决策，以最小化对冲刺的影响
2. **清晰沟通** - 确保所有利益相关者理解变更和影响
3. **记录化** - 记录所有变更决策用于审计和改进
4. **保持焦点** - 平衡灵活性和冲刺目标完整性

## 何时使用此工作流

- ✓ 冲刺期间出现新的需求或变更请求
- ✓ 发现导致实现方式改变的技术问题
- ✓ 优先级变更导致故事排序改变
- ✓ 需要重新评估设计或架构决策

## 何时不使用此工作流

- ✗ 在冲刺计划期间讨论故事优先级（使用冲刺规划工作流）
- ✗ 日常开发中的小的实现细节（直接与团队沟通）
- ✗ 简单的错误修复（使用标准的代码审查流程）
