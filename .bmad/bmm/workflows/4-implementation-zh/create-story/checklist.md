# 创建故事 - 故事背景质量竞争提示

## 故事背景验证框架

### 1. 故事头部

检查点:
- 故事 ID 和标题格式正确（例如：1.1: 用户认证）
- 故事状态设置为"已起草"
- 故事清晰地属于某个史诗

### 2. 用户故事格式

验证:
- [ ] 格式：作为 {{角色}}, 我想 {{行动}}, 以便 {{益处}}
- [ ] 角色清晰且有具体背景
- [ ] 行动具体且可测试
- [ ] 益处清晰且与业务相关

### 3. 接受标准

每个接受标准应该：
- [ ] 以"给定...当...那么..."或"应该..."的格式
- [ ] 是可测试的且明确的
- [ ] 独立且不重叠
- [ ] 覆盖正常流程和边界情况
- [ ] 至少有 3-5 个标准

### 4. 任务和子任务分解

验证:
- [ ] 任务清晰地对应接受标准
- [ ] 每个任务有 1-3 个子任务
- [ ] 子任务是可实现的工作项
- [ ] 列出了所有技术任务（设置、测试等）

### 5. 开发备注

应该包含:
- [ ] 相关的架构模式和约束
- [ ] 需要修改的源树组件
- [ ] 测试标准总结
- [ ] 项目结构说明

### 6. 上下文完整性

- [ ] 所有引用都有源路径和部分链接
- [ ] 技术细节有支持文档
- [ ] 没有模糊的引用或假设

### 7. 准备检查

- [ ] Dev Agent 能清晰理解故事
- [ ] 没有歧义或遗漏信息
- [ ] 团队可以开始实现
- [ ] 质量符合定义

## 故事状态生命周期

```
已起草 → 就绪待开发 → 进行中 → 审查中 → 完成
```

## 质量标准总结

| 方面 | 标准 | 检查 |
|-----|------|------|
| 清晰度 | 故事无歧义 | ✓ |
| 完整性 | 所有必要信息存在 | ✓ |
| 可测试性 | 所有接受标准可验证 | ✓ |
| 独立性 | 故事可以独立实现 | ✓ |
| 大小 | 故事适合一个冲刺 | ✓ |
