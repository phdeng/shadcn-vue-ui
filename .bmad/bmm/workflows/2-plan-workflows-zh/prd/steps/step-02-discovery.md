# 步骤 2：项目与领域发现

**进度：步骤 2 / 11** - 下一步：成功标准定义

**步骤目标**: 进行全面的项目发现，利用现有输入文档，同时允许用户细化，具有数据驱动的分类，并生成执行摘要内容。

## 强制执行规则

- 🛑 永不生成没有用户输入的内容
- 📖 关键：在采取任何行动前始终阅读完整步骤文件
- ✅ 始终将此视为产品经理同行之间的协作发现
- 📋 您是协调员，而不是内容生成器

## 执行序列

### 1. 从前置信息读取文档状态

从 `{outputFile}` 前置信息读取文档计数。

根据文档计数宣布您的理解：

- 产品简介：{{briefCount}} 个文件
- 研究：{{researchCount}} 个文件
- 头脑风暴：{{brainstormingCount}} 个文件
- 项目文档：{{projectDocsCount}} 个文件

这是 **棕地项目** 还是 **绿地项目**？

### 2. 加载分类数据

为智能分类加载并准备 CSV 数据：

- 加载 `{projectTypesCSV}`
- 加载 `{domainComplexityCSV}`
- 解析列结构并为此步骤存储在内存中

### 3. 开始发现对话

**根据文档状态选择确切的一个发现路径：**

#### 路径 A：有产品简介（briefCount > 0）

如果您有产品简介，阅读它并总结：

- 正在构建什么
- 它解决什么问题
- 目标用户
- 什么使它特殊

#### 路径 B：无简介但有项目文档 - 棕地（briefCount == 0 AND projectDocsCount > 0）

对于现有项目：

- 总结现有技术堆栈和架构
- 问用户他们想添加或更改什么
- 关注集成方式

#### 路径 C：无文档 - 绿地（briefCount == 0 AND projectDocsCount == 0）

对于新项目：

- 询问他们想构建什么
- 谁是他们正在为其构建的用户
- 什么使他们对此产品感到兴奋

### 4. 听取分类信号

当用户描述他们的产品/功能时，根据 CSV 数据听取并匹配：

- 项目类型信号（API、移动、Web 等）
- 领域信号（医疗保健、金融科技等）
- 复杂性级别

### 5. 呈现分类以验证

根据文档状态和对话呈现分类，问用户这听起来是否正确。

### 6. 识别什么使其特殊

- 如果有简介：从简介中提取差异化，讨论它如何演变
- 如果有项目文档：它如何填补现有系统中的空白
- 如果是绿地：什么使此产品与众不同/更好

### 7. 生成执行摘要内容

准备追加到文档的内容：

```markdown
## 执行摘要

{vision_alignment_content}

### 什么使此特殊

{product_differentiator_content}

## 项目分类

**技术类型：** {project_type}
**领域：** {domain}
**复杂性：** {complexity_level}
```

### 8. 呈现内容和菜单

显示已生成的内容并呈现：

"我已根据我们的对话起草了执行摘要。

**选择一个选项：**
[A] 高级启发 - 让我们深入并完善此内容
[P] 党派模式 - 引入不同的观点来改进
[C] 继续 - 保存并移到成功标准定义（步骤 3 / 11）"

### 9. 处理菜单选择

- **如果 A**：执行高级启发，询问用户是否接受改进
- **如果 P**：执行党派模式，询问用户是否接受更改
- **如果 C**：追加最终内容到 `{outputFile}` 更新前置信息 `stepsCompleted: [1, 2]` 加载 {nextStepFile}
