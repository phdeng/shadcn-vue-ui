# shadcn-vue-ui 核心需求（不可违反）

以下规则在项目的整个生命周期中永远生效，任何 AI Agent、任何开发任务、任何输出都不得违反。

---

## 一、语言规则

1. **永远使用中文输出** — 包括回答、注释、文档、说明、commit message
2. **代码命名使用英文** — 变量、函数、组件名遵循 Vue 社区英文命名规范
3. 不得混用中英文（除代码内嵌的英文术语外）

**Why:** 用户在首次 /init 中明确要求"永远使用中文输出、中文注释、中文说明、中文文档"，这是项目面向中文开发者社区的核心定位
**How to apply:** 每次输出前自检语言，每次写代码时注释用中文，每次写文档时全中文

---

## 二、项目身份（是什么 / 不是什么）

### 这个项目是：
1. 一个**现代化、轻量化、设计系统驱动**的 Vue 3 中后台前端框架
2. 一个面向**企业团队 + 创业团队 + 独立开发者**的中后台解决方案
3. 一个**双子项目体系**的框架（console + business）
4. 一个面向 **AI 协作时代**设计的项目
5. 一个基于 **shadcn 思想**（源码级可控）的组件体系

### 这个项目不是：
1. 不是传统"后台模板"（不是拼装页面交付了事）
2. 不是单一后台系统（是包含两个子项目的框架体系）
3. 不是重型 UI 框架（不依赖 Element Plus / Ant Design Vue 等）
4. 不是大而全的技术展示（拒绝过度工程化）

**Why:** 项目定位决定了所有后续决策的方向，偏离定位等于偏离项目
**How to apply:** 每次技术选型、组件设计、页面规划时，先对照这个身份清单，看是否偏离

---

## 三、双子项目体系（不可合并）

### console — AI 基础设施与平台能力管理
- 信息架构：**对象驱动**（模型、智能体、知识库、任务、服务）
- 交互重点：配置编排、状态观测、关联关系
- 视觉气质：专业、结构化、偏技术控制台

### business — 通用业务管理与运营后台
- 信息架构：**流程驱动**（用户、角色、订单、工单、内容）
- 交互重点：表格表单效率、筛选搜索、批量操作
- 视觉气质：清晰、稳重、偏业务管理

### 共享与独立的平衡
- **共享：** 设计 Token、基础 UI 组件、工具函数、工程规范
- **独立：** 场景组件、页面、路由、布局、Store

**Why:** 不是所有后台系统都适合用同一套模板，AI 平台和业务后台的产品范式本质不同
**How to apply:** 开发任何功能时必须明确它属于 shared / console / business 中的哪一个

---

## 四、六大核心设计原则（按优先级排序）

### 1. 轻量化优先
- 能不用重型依赖就不用
- 优先关注包体积、打包速度、运行性能
- 优先建设**最小可行架构**，再逐步增强
- 核心依赖控制在 10 个以内

### 2. 高审美优先
- **禁止**做成传统黑、白、蓝模板式后台
- 风格关键词：**简洁、克制、现代、留白、通透、精致、高级**
- 亮色 + 暗色双模式必须有统一设计语言
- 初创阶段每个子项目只维护 1 亮 + 1 暗主题

### 3. 设计系统优先
- 先建立设计语言 → 再沉淀组件 → 再构建页面
- 基础组件 → 场景组件 → 业务区块 → 页面模板，清晰分层
- 颜色、间距、圆角、阴影、字体、动效全部 Token 化

### 4. shadcn 思想优先
- 组件源码在项目内，不是 npm 黑盒
- 组件可理解、可复制、可组合、可扩展
- 封装透明，不做黑盒式封装

### 5. 场景化优先
- 不只做通用 UI 组件，重点建设管理后台高频场景组件
- Console 场景：模型卡片、配置面板、状态指示器、日志面板、用量仪表盘
- Business 场景：高级筛选栏、CRUD 模板、统计概览、审批流

### 6. AI 协作优先
- 每个模块职责清晰、边界明确、命名统一、文档可追踪
- 减少隐式约定，增加显式规则
- 所有页面/组件/模块都要易于被大模型理解和扩展

**Why:** 这 6 条原则是项目区别于其他 Vue 后台框架的根本，违反任何一条都会导致项目失去核心竞争力
**How to apply:** 每次设计决策时按优先级排序检查，轻量化 > 审美 > 设计系统 > shadcn > 场景化 > AI 协作

## 五、技术红线（绝对不可违反）

### 必须使用
- Vue 3 Composition API + `<script setup>`
- TypeScript 严格模式
- Vite 构建
- Tailwind CSS v4（oklch 色彩空间 + CSS 变量）
- Reka UI 组件原语
- shadcn-vue 源码级组件
- Vue Router 4
- Pinia
- Yarn workspace（Monorepo）
- Lucide Vue Next 图标

### 绝对禁止
- 禁止使用 Options API 或 mixins
- 禁止使用 Element Plus / Ant Design Vue / Naive UI 等重型 UI 库
- 禁止使用 Sass / Less / CSS-in-JS / UnoCSS
- 禁止使用 Vuex
- 禁止使用 Webpack
- 禁止使用 npm / pnpm（必须 Yarn）
- 禁止全量引入 lodash / moment.js / dayjs
- 禁止硬编码颜色值（必须用 CSS 变量）
- 禁止使用 `any` 类型
- 禁止引入微前端框架（当前阶段）
- 禁止引入 SSR / Nuxt（后台系统不需要）

**Why:** 技术栈已经过系统评估，每一项选择都服务于"轻量化 + 高质量 + AI 友好"的目标
**How to apply:** 引入任何新依赖前，先对照这个红线清单

---

## 六、AI 协作创新（核心卖点）

本项目必须在"AI 协作开发"方面形成核心创新，这套能力至少包含：

1. **项目认知入口** — 新模型接手项目时的标准阅读顺序
2. **需求拆解方法** — 复杂需求 → 页面 + 区块 + 组件 + 状态 + 接口 + 文档
3. **组件开发模板** — 统一格式，AI 稳定输出
4. **页面开发模板** — 统一结构，减少风格漂移
5. **任务卡机制** — 复杂需求拆成 AI 可执行的小任务
6. **文档反哺机制** — 每开发一个模块，同步更新文档
7. **自检机制** — 规范性、美观度、完整性、可维护性四维自检
8. **Prompt 规范** — 可复用的 Prompt 模板体系
9. **Skills 教学法** — 适合不同大模型的标准操作规范

**Why:** 这是 shadcn-vue-ui 区别于所有其他 Vue 后台框架的核心卖点
**How to apply:** 这 9 项能力必须在文档、Skills、工程结构中持续建设和完善

## 七、输出规范（每次回答必须遵守）

1. 先给结论，再给方案，再给可落地内容
2. 每次技术选型都必须说明"为什么"
3. 关键设计决策必须说明理由
4. 方案必须说明边界、优先级和后续演进路径
5. 如有多种方案，推荐最适合项目目标的一种
6. 所有输出都必须服务于本项目目标，不要跑题

**Why:** 确保输出质量一致，避免泛泛而谈
**How to apply:** 每次输出前自检是否符合这 6 条格式要求

AI 在本项目中不只是写代码，还必须同时承担：
- 项目定位规划
- 前端工程架构设计
- UI 设计系统规划
- 组件体系建设
- AI 协作开发方法设计
- 技术文档体系建设
- 官方推广文档策划

**Why:** 这是一个从 0 到 1 的完整开源项目，需要全方位的系统性规划
**How to apply:** 接到任务时，不只考虑代码实现，还要考虑对设计系统、文档体系、AI 协作方法的影响
