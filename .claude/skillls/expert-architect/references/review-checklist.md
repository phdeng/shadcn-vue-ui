# 审查清单

## 代码评审

### 致命问题（必须阻止合并）

**正确性**
- [ ] 逻辑是否正确？边界条件是否覆盖？
- [ ] 并发场景是否安全？竞态条件？死锁？
- [ ] 错误处理是否完备？异常是否会被吞掉？
- [ ] 资源是否正确释放？内存泄漏？连接泄漏？

**安全性**
- [ ] 输入是否验证？SQL注入？XSS？
- [ ] 敏感信息是否暴露？日志中有密码？
- [ ] 权限检查是否到位？越权访问？
- [ ] 依赖包是否有已知漏洞？

### 严重问题（应该阻止合并）

**设计问题**
- [ ] 是否过度设计？解决100%问题但复杂度提升10倍？
- [ ] 是否设计不足？关键场景没有考虑？
- [ ] 职责是否清晰？一个函数是否干了太多事？
- [ ] 依赖方向是否合理？是否引入循环依赖？

**可维护性**
- [ ] 命名是否清晰表达意图？
- [ ] 复杂逻辑是否有注释说明"为什么"？
- [ ] 魔法数字是否提取为常量？
- [ ] 重复代码是否应该抽取？

### 建议改进（不阻止合并）

- [ ] 性能是否有明显优化空间？
- [ ] 是否有更简洁的实现方式？
- [ ] 测试覆盖是否充分？
- [ ] 文档是否需要更新？

### 尖锐问题模板

```
"这段代码在 [X场景] 下会 [Y问题]。改成 [Z方案]。"

示例：
"这段代码在高并发下会重复创建订单。用数据库唯一索引+幂等key来防重。"
"这个catch吞掉了所有异常，线上出问题根本没法排查。至少要记日志。"
"这里N+1查询，1000条数据要查1001次数据库。用批量查询或JOIN。"
```

## 架构评审

### 业务匹配度

- [ ] 架构复杂度是否match业务复杂度？
- [ ] 当前阶段是否需要这个架构？
- [ ] 团队是否有能力驾驭这个架构？
- [ ] 这个架构的运维成本是否可接受？

**过度设计信号**：
- 用户量<10万却上了微服务
- 数据量<1亿却用分布式数据库
- 团队<10人却拆了20个服务
- 预期QPS 100却设计了支撑10万QPS的架构

**设计不足信号**：
- 已知瓶颈没有解决方案
- 单点故障没有备份
- 核心数据没有备份/容灾
- 关键路径没有降级方案

### 关键决策点

对于每个重要的架构决策，检查是否有：

- [ ] 明确的问题陈述（为什么需要这个决策）
- [ ] 考虑过的备选方案（至少2-3个）
- [ ] 清晰的trade-off分析
- [ ] 决策的适用边界（什么情况下会失效）
- [ ] 回退方案（如果选错了怎么办）

### 系统边界

- [ ] 服务边界是否基于业务能力划分？
- [ ] 每个服务是否有明确的owner？
- [ ] 服务间通信方式是否一致？
- [ ] 数据归属是否清晰？（避免数据共享）
- [ ] API契约是否稳定？版本策略？

### 非功能性需求

**可用性**
- [ ] 目标SLA是多少？有依据吗？
- [ ] 单点故障有哪些？如何消除？
- [ ] 故障检测时间是多少？
- [ ] 故障恢复时间是多少？
- [ ] 有没有chaos engineering计划？

**可扩展性**
- [ ] 预期增长曲线是什么？
- [ ] 哪些是瓶颈点？
- [ ] 扩展方式是垂直还是水平？
- [ ] 扩展的代价（时间、成本）是多少？

**安全性**
- [ ] 数据分级了吗？敏感数据如何保护？
- [ ] 访问控制模型是什么？
- [ ] 审计日志是否完备？
- [ ] 合规要求是否满足？

## AI系统评审

### 问题定义

- [ ] AI是否是解决这个问题的必要手段？
- [ ] 成功指标是什么？如何度量？
- [ ] 可接受的错误率是多少？
- [ ] 错误的代价是什么？（假阳性 vs 假阴性）

**危险信号**：
- "我们要用AI" → 但不清楚解决什么问题
- "我们要提升用户体验" → 但没有具体指标
- "这是CEO要的" → 但没有技术可行性分析

### 数据Pipeline

- [ ] 数据源是否可靠？质量如何？
- [ ] 数据量是否足够？
- [ ] 数据标注质量如何保证？
- [ ] 数据更新频率是多少？
- [ ] 特征工程是否可复现？
- [ ] 训练数据和线上数据分布是否一致？

**常见问题**：
- 训练数据有偏差
- 数据泄露（用了不该用的特征）
- 线上线下特征计算不一致
- 数据标注标准不统一

### 模型选型

- [ ] 为什么选择这个模型？
- [ ] 有没有更简单的baseline？效果差多少？
- [ ] 模型复杂度与问题复杂度是否匹配？
- [ ] 推理延迟和吞吐量是否满足要求？
- [ ] 模型大小和部署资源是否可接受？

**选型原则**：
1. 先用规则系统
2. 规则不够用，试简单ML模型
3. 简单模型不够，再上深度学习
4. 深度学习不够，最后考虑大模型

### 上线与运维

- [ ] A/B测试方案是什么？
- [ ] 灰度发布策略是什么？
- [ ] 模型监控指标有哪些？
- [ ] 数据漂移如何检测？
- [ ] 模型退化如何处理？
- [ ] 人工兜底机制是什么？
- [ ] 降级策略是什么？

**必须有的监控**：
- 推理延迟（P50、P99）
- 模型输出分布
- 关键业务指标（CTR、转化率等）
- 异常case采样
- 用户反馈收集

### 成本评估

- [ ] 训练成本（算力、时间、人力）
- [ ] 推理成本（QPS、机器数、GPU/CPU）
- [ ] 数据成本（采集、标注、存储）
- [ ] 维护成本（迭代频率、团队配置）
- [ ] ROI是否合理？
