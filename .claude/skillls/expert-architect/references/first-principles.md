# 第一性原理分析框架

## 核心方法论

第一性原理思维的本质：**把问题拆解到最基本的事实和约束，从零开始推导解决方案，而非类比已有方案**。

### 三步分解法

**Step 1: 剥离表象，找到本质问题**

```
表象问题 → 追问 "为什么需要这个" → 本质需求
```

示例：
- 表象："我们需要引入消息队列"
- 追问：为什么需要消息队列？
- 本质：需要解耦两个系统的处理速度差异
- 再追问：真的需要解耦吗？流量峰值多少？
- 可能的真相：峰值QPS才200，同步处理完全够用

**Step 2: 识别真实约束**

```
物理约束（不可改变）
├── 网络延迟下限
├── 磁盘IO速度上限
├── 用户反应时间阈值
└── 资金/人力硬限制

假性约束（可以挑战）
├── "公司一直这么做"
├── "业界标准是X"
├── "领导说要用Y技术"
└── "文档上说不支持"
```

**Step 3: 最小充分解**

在满足真实约束的前提下，找到最简单的解决方案。

评估标准：
- 概念数量：引入了多少新概念？
- 移动部件：系统中有多少可能出故障的组件？
- 认知负担：新人需要多久能理解这个系统？

## 常见第一性原理问题

### 分布式系统

| 表面需求 | 第一性原理追问 | 常见结论 |
|---------|--------------|---------|
| "需要微服务" | 你们有多少人？系统多复杂？ | 10人以下团队99%不需要 |
| "需要Kubernetes" | 你们有多少服务？运维能力如何？ | 服务<10个，直接用docker-compose |
| "需要分布式数据库" | 数据量多大？读写QPS多少？ | 单机MySQL支撑到亿级数据 |
| "需要缓存" | 真正的瓶颈是什么？查询优化了吗？ | 80%的缓存需求是掩盖数据库设计问题 |

### AI/ML系统

| 表面需求 | 第一性原理追问 | 常见结论 |
|---------|--------------|---------|
| "需要用大模型" | 具体要解决什么问题？准确率要求？ | 很多场景规则引擎+小模型足够 |
| "需要实时推理" | 用户能接受多少延迟？批处理行不行？ | 很多"实时"需求其实可以接受秒级延迟 |
| "需要自己训练模型" | 有多少数据？有ML团队吗？ | 先用预训练模型+微调，不要从头训练 |
| "需要最新的模型架构" | 问题复杂度真的需要吗？ | BERT/RoBERTa解决90%的NLP问题 |

### 架构设计

| 表面需求 | 第一性原理追问 | 常见结论 |
|---------|--------------|---------|
| "需要高可用" | 具体要几个9？成本预算多少？ | 99.9%和99.99%成本差10倍 |
| "需要可扩展" | 预期增长曲线是什么？ | 为10x设计，100x重构，别为1000x设计 |
| "需要解耦" | 这两个模块真的会独立变化吗？ | 过早解耦比耦合更危险 |
| "需要抽象层" | 抽象是为了复用还是为了好看？ | 没有复用场景的抽象是负资产 |

## 分析模板

对任何技术决策，完成以下分析：

```markdown
## 决策分析：[决策名称]

### 本质需求
[用一句话描述真正要解决的问题]

### 真实约束
- 物理约束：[列出不可改变的限制]
- 业务约束：[列出业务层面的限制]
- 资源约束：[时间、人力、预算]

### 方案评估
| 方案 | 复杂度 | 满足需求 | 主要风险 |
|-----|-------|---------|---------|
| 方案A | 低 | 80% | xxx |
| 方案B | 高 | 100% | yyy |

### 建议
[明确选择某方案，说明理由]

### 什么时候这个选择会错
[列出导致决策失效的条件变化]
```

## 危险信号

以下说法出现时，要高度警惕：

- "大厂都在用" → 大厂的问题和你的问题一样吗？
- "未来可能需要" → 预测未来的准确率很低，YAGNI原则
- "这是最佳实践" → 在什么上下文下是最佳？
- "技术上更优雅" → 优雅不能当饭吃，简单直接更重要
- "方便以后扩展" → 等真的需要扩展时再扩展
