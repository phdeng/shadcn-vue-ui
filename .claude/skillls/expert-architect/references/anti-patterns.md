# 反模式识别指南

## 架构反模式

### 分布式单体 (Distributed Monolith)

**症状**：
- 服务拆成了10+个，但必须一起部署
- 改一个服务要同时改3个其他服务
- 服务间大量同步调用，形成长链路
- 一个服务挂了，整个系统不可用

**根因**：按技术层而非业务能力拆分服务

**诊断问题**：
- "你们能独立部署单个服务吗？"
- "改动一个功能平均涉及几个服务？"
- "服务A能脱离服务B独立运行吗？"

**解法**：重新按业务领域划分边界，或者退回单体

---

### CV架构 (Copy-paste Architecture)

**症状**：
- "某大厂用了这个架构，我们也用"
- 架构图画得很漂亮，但说不清为什么这样设计
- 团队不理解架构的关键组件为什么存在

**根因**：缺乏独立思考，用别人的解决方案解决自己的问题

**诊断问题**：
- "这个组件解决什么问题？"
- "如果去掉这个组件会怎样？"
- "这个设计的trade-off是什么？"

**解法**：从第一性原理出发，基于自己的约束设计架构

---

### 简历驱动开发 (Resume-Driven Development)

**症状**：
- 技术选型依据是"想学学这个"
- 系统里有大量未被使用的"基础设施"
- 文档里充满了高大上的技术词汇

**根因**：技术决策为个人成长而非业务需求服务

**诊断问题**：
- "去掉这个组件对业务有什么影响？"
- "这个技术选型的业务收益是什么？"
- "如果用更简单的方案，会损失什么？"

**解法**：建立以业务价值为核心的技术决策流程

---

### 过早优化 (Premature Optimization)

**症状**：
- 还没有用户就在考虑如何支撑百万并发
- 大量缓存层，但系统QPS只有个位数
- 复杂的异步架构处理简单的同步逻辑

**根因**：恐惧驱动的设计，而非需求驱动

**诊断问题**：
- "当前的性能瓶颈是什么？有数据吗？"
- "如果不做这个优化，会发生什么？"
- "这个优化的维护成本是多少？"

**解法**：先跑起来，用数据说话，优化真正的瓶颈

---

### 金 plated（镀金）

**症状**：
- 需求是简单CRUD，实现了完整的CQRS+Event Sourcing
- 一个配置中心，支持了15种配置格式
- 每个API都有5层抽象

**根因**：对"正确性"的过度追求，忽视ROI

**诊断问题**：
- "这个抽象层被复用了几次？"
- "去掉这层抽象，代码会变难维护吗？"
- "这个功能的使用频率是多少？"

**解法**：YAGNI - 只实现当前需要的功能

---

## 代码反模式

### 上帝类 (God Class)

**症状**：
- 一个类超过2000行
- 一个类有50+个方法
- 任何改动都要改这个类

**诊断**：`wc -l *.java | sort -n | tail -10`

**解法**：按职责拆分，单一职责原则

---

### 魔法数字/字符串

**症状**：
```java
if (status == 3) // 3是什么？
sleep(86400000) // 这是多久？
url = "http://192.168.1.100:8080" // 硬编码地址
```

**解法**：提取常量，命名要表达意图

---

### 异常吞噬

**症状**：
```java
try {
    doSomething();
} catch (Exception e) {
    // ignore
}
```

**危害**：出问题了无法排查

**解法**：至少记录日志，最好向上抛出

---

### 过度抽象

**症状**：
- 一个简单操作要跳转7个文件
- 接口只有一个实现
- 设计模式堆叠使用

**诊断问题**：
- "这个接口有几个实现？"
- "这层抽象解决了什么问题？"

**解法**：删除不必要的抽象层

---

### 注释代码

**症状**：
```java
// if (oldLogic) {
//     doOldThing();
// }
```

**危害**：代码库逐渐变成垃圾场

**解法**：删除。版本控制会记住历史。

---

## 流程反模式

### 英雄文化

**症状**：
- 只有某个人能改某个系统
- 上线必须某个人在场
- 出问题只能等某个人处理

**风险**：bus factor = 1

**解法**：
- 强制知识共享（pair programming、文档）
- 轮值制度
- 自动化减少人工依赖

---

### 流程债务

**症状**：
- "这个步骤很麻烦但一直这样做"
- 大量手工操作，容易出错
- 部署要执行20个步骤的checklist

**解法**：
- 自动化一切可以自动化的
- 定期review流程，删除不必要的步骤

---

### 无文档黑洞

**症状**：
- 系统设计只存在于某人脑子里
- 新人入职靠口口相传
- 出问题要考古代码

**解法**：
- ADR (Architecture Decision Record)
- 系统文档放在代码库里，和代码一起维护
- 入职文档模板化

---

## 组织反模式

### 康威定律的诅咒

**症状**：
- 系统架构完美复制了组织架构
- 团队边界 = 服务边界，无论是否合理
- 跨团队协作成本极高

**解法**：
- 按业务领域而非部门组建团队
- 跨职能团队（全栈团队）
- 明确服务owner而非代码owner

---

### 指标地狱

**症状**：
- KPI驱动的技术决策
- 为了完成指标而做技术债
- 代码覆盖率90%但测试都是无效的

**解法**：
- 关注结果指标而非行为指标
- 接受不可量化的工作
- 长期指标 vs 短期指标平衡

---

## 诊断技巧

### 快速识别问题的问题清单

1. "你们最近一次生产事故是什么？根因是什么？"
2. "部署一次需要多长时间？涉及多少人？"
3. "新人入职到能独立开发需要多久？"
4. "如果要加一个简单功能，需要改几个服务？"
5. "你们的技术债务有列表吗？谁在跟进？"
6. "上次做架构决策的文档在哪里？"

### 代码层面快速诊断

```bash
# 找最大的文件（可能是上帝类）
find . -name "*.java" -exec wc -l {} \; | sort -n | tail -20

# 找最多依赖的文件
grep -r "import" --include="*.java" | cut -d: -f2 | sort | uniq -c | sort -n | tail -20

# 找被注释的代码
grep -rn "//.*{" --include="*.java" | head -20

# 找硬编码的配置
grep -rn "192.168\|localhost\|8080" --include="*.java"
```
