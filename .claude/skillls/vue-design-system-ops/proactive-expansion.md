# 主动扩展检查清单

当用户只提出局部需求时，按此清单检查是否有需要同步处理的关联工作。

## 必做项（直接纳入任务）

### 类型安全
- [ ] 新增/修改的代码是否有完整 TypeScript 类型
- [ ] 新增的 Props 是否在 types.ts 中导出
- [ ] 是否引入了 `any` 类型

### API 一致性
- [ ] 新 API 是否与同类组件命名风格一致
- [ ] variant/size 等枚举值是否与项目约定一致
- [ ] emits 命名是否遵循项目模式

### 可访问性
- [ ] 交互元素是否有键盘支持
- [ ] 是否有缺失的 ARIA 属性
- [ ] 焦点管理是否正确

### 暗色模式
- [ ] 变更涉及的样式在暗色模式下是否正常
- [ ] 是否使用了 design token 而非硬编码颜色

---

## 建议项（列出建议，由用户决定）

### 文档
- [ ] README 是否需要更新
- [ ] 组件文档是否需要补充示例
- [ ] CHANGELOG 是否需要新增条目

### 测试
- [ ] 变更是否有对应测试
- [ ] 现有测试是否需要更新
- [ ] 是否需要新的边界条件测试

### 依赖影响
- [ ] 修改的模块被哪些组件/页面引用
- [ ] 上游变更是否需要下游适配
- [ ] monorepo 中其他包是否受影响

### 抽象机会
- [ ] 本次变更中是否有可复用的模式
- [ ] 是否有类似代码已存在于其他组件（可合并）
- [ ] 是否适合提取为 composable

### 设计系统一致性
- [ ] 新增的交互模式是否已有先例可参考
- [ ] 是否无意中引入了新的设计模式（应讨论）
- [ ] token 使用是否与全局体系一致

---

## 如何使用此清单

1. 在开始实现前快速过一遍必做项
2. 在实现完成后检查建议项
3. 必做项中发现的问题直接纳入当前任务
4. 建议项以"扩展建议"形式输出给用户
5. 用户确认后再执行建议项
