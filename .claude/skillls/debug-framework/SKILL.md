# 系统化调试框架

遵循 **调查 → 分析 → 测试 → 实施** 四步法进行系统化问题排查。

## 四步调试法概览

```
┌─────────────────────────────────────────────────────────────────┐
│  调查 (Investigate)  →  分析 (Analyze)  →  测试 (Test)  →  实施 (Implement)  │
│      收集信息              定位根因           验证假设          修复并验证      │
└─────────────────────────────────────────────────────────────────┘
```

## 第一步：调查 (Investigate)

**目标**：全面收集问题信息，建立问题画像。

### 收集清单

| 维度 | 关键问题 | 获取方式 |
|-----|---------|---------|
| 现象 | 具体错误信息是什么？ | 查看日志、控制台、错误堆栈 |
| 时机 | 什么时候开始出现的？ | 部署记录、git log、监控告警 |
| 频率 | 必现还是偶发？ | 多次复现尝试 |
| 范围 | 影响哪些功能/用户？ | 生产监控、用户反馈 |
| 环境 | 哪些环境有问题？ | 对比开发/测试/生产环境 |
| 变更 | 最近有什么改动？ | 代码diff、配置变更、依赖更新 |

### 调查命令速查

```bash
# 查看最近代码变更
git log --oneline -20
git diff HEAD~5

# 查看错误日志
tail -f /var/log/app.log | grep -i error
journalctl -u service-name --since "1 hour ago"

# 查看系统资源
top -bn1 | head -20
df -h
free -m

# 查看网络连接
netstat -tlnp
ss -tulpn
```

### 调查输出模板

```markdown
## 问题画像

**现象描述**：[具体错误信息和表现]

**复现条件**：
- 环境：[开发/测试/生产]
- 步骤：1. ... 2. ... 3. ...
- 频率：[必现/偶发，概率约X%]

**影响范围**：[受影响的功能、用户数量]

**时间线**：
- 首次发现：[时间]
- 最近变更：[变更内容和时间]

**已收集信息**：
- 错误日志：[关键日志片段]
- 堆栈信息：[调用栈]
- 环境信息：[版本、配置等]
```

## 第二步：分析 (Analyze)

**目标**：基于收集的信息，系统性定位根本原因。

### 假设生成

根据问题类型生成假设列表：

| 问题类型 | 常见根因方向 |
|---------|------------|
| 运行时错误 | 空指针、类型错误、越界访问、资源未释放 |
| 性能问题 | N+1查询、内存泄漏、死锁、资源争用 |
| 逻辑错误 | 边界条件、竞态条件、状态不一致 |
| 集成问题 | 接口变更、超时配置、认证失效、版本不兼容 |
| 环境问题 | 配置差异、依赖缺失、权限不足 |

### 分析方法

**二分法定位**：
```
正常版本 ←──────────────────────→ 问题版本
           逐步缩小范围定位引入点
```

```bash
# Git 二分查找
git bisect start
git bisect bad HEAD
git bisect good v1.0.0
# 测试后标记 git bisect good/bad
```

**排除法**：
```
假设列表 → 设计验证方法 → 逐个排除 → 锁定根因
    A          验证A           排除A
    B          验证B           确认B ✓
    C          验证C           排除C
```

**堆栈分析**：
- 从错误位置向上追溯调用链
- 关注参数传递和状态变化
- 检查异常捕获和处理逻辑

### 分析输出模板

```markdown
## 根因分析

**假设列表**（按可能性排序）：
1. [假设A] - 可能性：高 - 依据：[分析理由]
2. [假设B] - 可能性：中 - 依据：[分析理由]
3. [假设C] - 可能性：低 - 依据：[分析理由]

**排除记录**：
- ❌ [假设C]：[排除依据]
- ❌ [假设A]：[排除依据]

**根因定位**：
- ✅ [假设B]：[确认依据]
- 根本原因：[详细描述]
- 影响机制：[问题如何导致现象]
```

## 第三步：测试 (Test)

**目标**：验证分析结论，确保修复方案有效。

### 验证策略

| 验证类型 | 方法 | 适用场景 |
|---------|-----|---------|
| 最小复现 | 编写最简测试用例 | 逻辑错误、边界条件 |
| 对照实验 | 控制变量对比 | 环境问题、配置问题 |
| 日志注入 | 添加详细日志 | 流程问题、状态追踪 |
| 断点调试 | 单步执行观察 | 复杂逻辑、变量状态 |
| 压力测试 | 并发/大数据量 | 性能问题、竞态条件 |

### 测试用例设计

```python
# 最小复现测试模板
def test_bug_reproduction():
    """
    Bug: [简述问题]
    根因: [分析结论]
    预期: 修复后应该[预期行为]
    """
    # 1. 设置最小必要条件
    setup_minimal_context()

    # 2. 触发问题条件
    result = trigger_bug_condition()

    # 3. 验证问题存在（修复前应失败）
    assert result == expected_behavior
```

### 修复验证清单

- [ ] 问题是否可复现？
- [ ] 修复后问题是否消失？
- [ ] 相关功能是否正常？
- [ ] 是否引入新问题？
- [ ] 边界情况是否处理？

### 测试输出模板

```markdown
## 验证报告

**复现测试**：
- 测试用例：[描述]
- 执行结果：✅ 成功复现问题

**修复验证**：
- 修复方案：[描述]
- 验证结果：✅ 问题已解决

**回归测试**：
- 相关功能：✅ 正常
- 边界情况：✅ 已覆盖
- 新增风险：无
```

## 第四步：实施 (Implement)

**目标**：安全地实施修复并确保不引入新问题。

### 修复原则

1. **最小改动**：只修改必要部分，降低引入新问题风险
2. **可回滚**：确保可以快速回滚到修复前状态
3. **可验证**：修复效果可被自动化测试验证
4. **有文档**：记录问题原因和修复方案

### 实施流程

```
代码修复 → 本地测试 → Code Review → 测试环境 → 生产环境 → 监控确认
                                                    ↓
                                              问题复发？→ 回滚
```

### 代码修复模板

```python
# 修复前代码（保留注释说明问题）
# BUG: 当 items 为空时会导致 IndexError
# result = items[0]

# 修复后代码
# FIX: 添加空列表检查，返回 None 作为默认值
result = items[0] if items else None
```

### Commit Message 规范

```
fix(module): 简短描述修复内容

问题：[描述问题现象]
根因：[说明根本原因]
方案：[解释修复方法]

Fixes #123
```

```markdown
## 修复报告

**修复方案**：
- 修改文件：[文件列表]
- 改动说明：[具体改动]

**部署记录**：
- 部署时间：[时间]
- 部署版本：[版本号]
- 回滚方案：[如何回滚]

**验证确认**：
- 生产验证：✅ 问题已解决
- 监控状态：✅ 无异常告警
- 用户反馈：✅ 功能正常

**经验总结**：
- 问题根因：[一句话总结]
- 预防措施：[如何避免类似问题]
```

## 快速诊断指南

### 常见错误速查

| 错误类型 | 典型表现 | 快速检查 |
|---------|---------|---------|
| NullPointer | undefined/null 访问 | 检查数据来源和初始化 |
| TypeError | 类型不匹配 | 检查参数类型和转换 |
| Timeout | 请求超时 | 检查网络、服务状态、超时配置 |
| OOM | 内存溢出 | 检查内存泄漏、大对象 |
| Deadlock | 程序卡死 | 检查锁顺序、资源等待 |
| 404/500 | HTTP错误 | 检查路由、服务日志 |

### 调试模式选择

**快速模式**（5-15分钟）：
- 适用：明显错误、有清晰堆栈
- 流程：查日志 → 定位代码 → 直接修复

**标准模式**（30-60分钟）：
- 适用：需要复现和分析
- 流程：完整四步法

**深度模式**（数小时）：
- 适用：复杂问题、偶发问题
- 流程：四步法 + 详细记录 + 多轮迭代
