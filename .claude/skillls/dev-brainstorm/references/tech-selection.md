# 技术选型对比框架

## 通用对比维度

### 核心维度

| 维度 | 评估要点 | 权重参考 |
|-----|---------|---------|
| 功能完备性 | 是否满足当前和可预见的需求 | 高 |
| 性能表现 | 吞吐量、延迟、资源占用 | 视场景 |
| 稳定性 | 生产环境验证程度、已知问题 | 高 |
| 可维护性 | 代码质量、文档、调试便利性 | 中 |
| 扩展性 | 水平/垂直扩展能力、插件机制 | 视场景 |

### 生态维度

| 维度 | 评估要点 |
|-----|---------|
| 社区活跃度 | GitHub stars、PR 频率、issue 响应 |
| 文档质量 | 官方文档完整性、教程丰富度 |
| 商业支持 | 是否有企业版、付费支持 |
| 人才市场 | 招聘难度、学习资源 |

### 成本维度

| 维度 | 评估要点 |
|-----|---------|
| 授权费用 | 开源/商业、按量/按席位 |
| 学习成本 | 团队熟悉度、上手难度 |
| 运维成本 | 部署复杂度、监控需求 |
| 迁移成本 | 锁定程度、替换难度 |

## 场景化选型指南

### 数据库选型

```
需求分析:
├─ 数据模型 → 关系型/文档型/KV/图
├─ 一致性要求 → 强一致/最终一致
├─ 读写比例 → 读多写少/写多读少
├─ 数据规模 → 单机/分布式
└─ 查询复杂度 → 简单查询/复杂分析
```

常见选择：
- **关系型数据，ACID 需求**：PostgreSQL, MySQL
- **文档存储，灵活 schema**：MongoDB
- **高并发缓存/会话**：Redis
- **大规模分析**：ClickHouse, Elasticsearch
- **时序数据**：TimescaleDB, InfluxDB

### 消息队列选型

```
需求分析:
├─ 消息量级 → 低(<1K/s)/中/高(>100K/s)
├─ 可靠性要求 → 允许丢失/至少一次/精确一次
├─ 消费模式 → 点对点/发布订阅
├─ 消息顺序 → 全局有序/分区有序/无序
└─ 延迟要求 → 实时(<10ms)/准实时(<1s)/可容忍
```

常见选择：
- **通用场景，生态完善**：Kafka
- **轻量级，协议标准**：RabbitMQ
- **云原生，Serverless**：各云厂商产品
- **高性能，低延迟**：Pulsar

### Web 框架选型

```
需求分析:
├─ 项目类型 → API 服务/全栈应用/微服务
├─ 团队技术栈 → 现有语言经验
├─ 性能要求 → CPU 密集/IO 密集
├─ 开发效率 → 快速迭代/长期维护
└─ 部署环境 → 容器/Serverless/传统服务器
```

## 红旗警告

以下情况需要特别谨慎：

- ⚠️ 最近一年无重大更新
- ⚠️ 核心维护者离开项目
- ⚠️ 存在未修复的严重安全漏洞
- ⚠️ 许可证变更风险（如 Elastic, MongoDB）
- ⚠️ 唯一供应商依赖
- ⚠️ 团队完全没有相关经验且时间紧迫
