# 开发头脑风暴 Skill

帮助开发者进行结构化的技术思考和方案探索。

## 核心工作流程

### 1. 明确问题域

首先理解用户真正想解决的问题：

- **目标**：用户想达成什么？
- **约束**：时间、资源、技术栈、团队能力
- **上下文**：现有系统、历史决策、业务背景

提问示例：
- "这个功能主要解决什么业务问题？"
- "有什么技术或资源上的限制吗？"
- "需要和现有哪些系统集成？"

### 2. 发散思考

根据问题类型选择合适的发散策略：

| 问题类型 | 发散策略 |
|---------|---------|
| 新功能设计 | 用户场景枚举 → 功能点推导 → 实现方案 |
| 技术选型 | 候选方案列举 → 多维度对比 → 权衡分析 |
| 架构设计 | 职责拆分 → 模块划分 → 交互方式 |
| 问题排查 | 假设列举 → 验证方法 → 排除策略 |
| 性能优化 | 瓶颈定位 → 优化手段 → 收益评估 |

### 3. 收敛决策

将发散的想法收敛为可执行的方案：

```
发散结果 → 可行性评估 → 优先级排序 → 推荐方案
```

输出包含：
- **推荐方案**：最佳选择及理由
- **备选方案**：在特定条件下的替代选择
- **风险提示**：潜在问题和应对策略
- **下一步行动**：具体的执行步骤

## 头脑风暴模式

### 模式一：快速探索（5分钟）

适用于简单问题或初步方向确认。

```
问题 → 3-5个方向 → 简要对比 → 推荐
```

### 模式二：深度分析（15-30分钟）

适用于复杂决策或重要架构设计。

```
问题定义 → 全面发散 → 详细对比 → 方案设计 → 风险分析
```

### 模式三：引导式对话

适用于问题不明确或需要逐步探索。

通过提问引导用户思考：
- "如果不考虑成本，理想方案是什么？"
- "最坏情况下会发生什么？"
- "有没有类似问题的解决经验？"

## 常用思考框架

### 技术选型五维评估

1. **功能适配度**：能否满足需求？
2. **性能表现**：能否满足性能要求？
3. **开发效率**：学习成本和开发速度？
4. **运维成本**：部署、监控、维护难度？
5. **生态成熟度**：社区活跃度、文档质量？

### 架构设计原则

- **单一职责**：每个模块只做一件事
- **高内聚低耦合**：相关功能聚合，减少依赖
- **可测试性**：易于编写单元测试
- **可扩展性**：预留扩展点，避免过度设计

### 问题排查漏斗

```
现象描述 → 复现条件 → 影响范围 → 假设列表 → 验证排除 → 根因定位
```

## 提示词模板

当用户需求模糊时，使用以下模板引导：

**新功能设计**
> 让我们来梳理一下：
> 1. 这个功能的目标用户是谁？
> 2. 用户的核心使用场景是什么？
> 3. 功能的边界在哪里？什么不做？

**技术选型**
> 在做选型前，先确认几个关键点：
> 1. 必须满足的硬性要求是什么？
> 2. 团队对哪些技术比较熟悉？
> 3. 预期的规模和性能要求？

**问题排查**
> 让我们系统地分析：
> 1. 问题最早什么时候出现的？
> 2. 能稳定复现吗？复现条件是什么？
> 3. 最近有什么变更？代码、配置、环境？
